# RepoPilot 核心知识与面试指南

> 本文档整理 RepoPilot 项目涉及的核心技术知识点、架构决策和面试高频问题的完整回答。
> 目标：面试时能完整、清晰、有深度地讲清楚整个项目。

---

## 一、项目概述（30秒电梯演讲）

**RepoPilot 是一个本地优先的 Code Agent CLI 工具**，类似 Claude Code / Codex CLI。用户在终端输入自然语言任务（如"修复login函数的bug"、"给这个模块加单元测试"），Agent 自动分析代码库、调用工具（读文件、搜索、执行命令、编辑代码）、在沙盒中验证，最终完成任务。

**核心特点：**
- **本地优先**：单租户场景，不依赖云端服务，用户自带 API Key
- **Docker 沙盒隔离**：代码在容器中执行，有CPU/内存/网络限制，防止恶意代码破坏宿主
- **三层模型路由**：fast模型做规划反思（省成本），default模型做编码，strong模型做最终输出
- **生产级稳定性**：熔断器、指数退避重试、路径穿越防护、超时控制、token预算
- **不使用RAG/向量数据库**：代码检索通过grep/glob/tree-sitter等确定性工具完成，更精准无需embedding

## 二、面试高频问题与完整回答

### Q1: 介绍一下你的 Code Agent 项目，整体架构是怎样的？

**回答框架：**

RepoPilot 采用分层架构，从下到上分为四层：

**1. 配置层（Config）**
- 统一管理所有配置：模型选择、API Key、沙盒参数、权限模式、Agent参数
- 四层优先级：默认值 < TOML配置文件 < 环境变量 > 命令行参数
- 首次运行自动引导用户配置，配置持久化到 `~/.repopilot/config.toml`

**2. LLM 服务层（LLM Service）**
- 基于 LiteLLM 统一调用各种模型（火山ARK/DeepSeek/GLM/OpenAI）
- 三层模型路由：
  - **Fast tier**（15s超时）：用便宜小模型做规划、反思、上下文压缩
  - **Default tier**（25s超时）：主力模型，处理主要的编码任务，用5M上下文的doubao-seed-evolving
  - **Strong tier**（40s超时）：高质量模型做最终答案生成
- 生产级容错：指数退避+全抖动重试（可重试错误包括超时/429/5xx），滑动窗口熔断器（错误率超50%触发熔断60秒冷却）

**3. 沙盒层（Sandbox）**
- 定义统一抽象接口：read_file/write_file/edit_file/exec/glob/grep/list_dir/get_repo_tree
- 两种实现可互换：
  - **LocalSandbox**：直接操作本地文件系统，开发调试用，有路径穿越防护
  - **DockerSandbox**：真正隔离，bind mount仓库到容器，限制2核CPU/2G内存/网络模式可控
- 所有工具调用都经过沙盒层，Agent不直接接触文件系统

**4. 代码索引层（Code Index / Repo Map）**（T11实现）
- tree-sitter AST解析：支持Python，提取class/function/method/imports，包含行号和docstring
- 忽略机制：自动跳过.git、__pycache__、node_modules、.min.js、二进制文件、.env等，支持.gitignore（pathspec）
- Token预算控制：~4 chars/token近似，小文件优先排序（大文件通常是依赖或生成代码），超过预算截断并显示省略数
- 缓存机制：文件编辑后调用builder.update(path)失效单个文件缓存，无需全量重建
- LocalSandbox.get_repo_tree()和DockerSandbox.get_repo_tree()都已接入RepoMapBuilder，失败时fallback到简单文件列表
- 输出格式示例：`main.py` → `  L1 def add() — Add two numbers`、`  L5 class Calculator — A simple calculator`

**5. 会话存储层（Session Store）**（T12实现）
- JSONL追加日志：每行一个事件`{"id": N, "ts": ISO8601, "type": "...", "payload": {...}}`，路径`~/.repopilot/sessions/YYYY/MM/DD/rollout-<ts>-<id>.jsonl`
- SQLite索引：`~/.repopilot/state.sqlite`的threads表存储会话元数据（id/title/cwd/model/created_at/updated_at/tokens_used/last_event_id），毫秒级时间戳
- 支持12种事件类型：user_msg/assistant_msg/tool_call/tool_result/plan/replan/finish/error/slash/system/compact/cost
- 核心方法：create/get/list/delete/append_event/read_events/rewind/update_title
- rewind()回退功能：截断到第N步，支持undo/retry场景（用户对Agent的错误操作不满意时回退重试）
- tokens_used自动累计：从assistant_msg的usage.total_tokens字段自动累加
- 线程安全：per-session threading.Lock，原子追加写入

**6. 上下文管理层（Context Manager）**（T13实现）
- 分层消息组装：L0 system prompt → L0.5 repo_map+memory+skills → plan段 → L2 compacted summary → L1 recent steps → pending tool result → task
- 三级压缩体系：
  - **tool-compact（规则，无LLM）**：工具输出超过12000字符/200行时，保留head+tail中间显示省略，纯规则不调LLM
  - **micro-compact（fast LLM，75%阈值触发）**：用fast模型将最老的3-5步压缩为1-2句话，保留核心信息
  - **auto-compact（fast LLM，90%阈值触发）**：用fast模型将除最近10步外的所有历史压缩为结构化摘要（Completed Actions/Key Findings/File Locations/Unresolved四部分）
- Token估算：len(text)//4近似，拿到真实API usage后通过update_actual_usage()校准
- 预算感知：build_messages时_select_steps_for_budget()从最新消息开始倒序放入，保证最近信息优先保留
- 关键阈值：SYSTEM_RESERVE=1000, PROJECT_RESERVE=5000, PLAN_RESERVE=1000, SUMMARY_RESERVE=2000, RECENT_STEPS_KEEP=10

**4. Agent 层（Agent Loop）**（核心，Step 7实现）
- ReAct循环：Thought → Action（Tool Call）→ Observation → 循环直到任务完成
- 每一步将对话历史+工具结果发给LLM，LLM决定是继续调工具还是给出最终答案
- 有max_steps（50步）和token预算（200K）防止无限循环和成本失控
- 上下文自动压缩：对话超过阈值时用fast模型压缩历史
- 权限控制：危险操作（写文件/执行命令）根据approval_mode决定是否需要用户确认

**关键设计决策：为什么不用RAG？**
- Code Agent是单租户场景（一个用户操作自己的代码库），代码在本地文件系统直接可访问
- 代码检索用grep/glob/tree-sitter是确定性的、精准的，不需要embedding近似检索
- 向量检索会有幻觉和遗漏，grep不会——grep要么找到要么没找到
- 这也是Codex CLI和Claude Code的做法，它们也不用RAG

### Q2: Agent 的记忆管理怎么做？处理流程是什么？

**当前实现（会话内记忆）+ 规划（跨会话记忆）：**

**会话内记忆（Context Window）：**
- 维护一个messages列表：system prompt → user task → assistant（tool_calls）→ tool_results → assistant → ...
- 每个tool_result包含工具名称、参数和返回结果
- 控制策略：
  - `recent_keep_steps=10`：压缩时保留最近10轮完整对话
  - `compact_micro_ratio=0.7`：上下文使用70%时触发轻量压缩
  - `compact_auto_ratio=0.85`：上下文使用85%时触发强制压缩
  - 压缩方式：用fast模型对早期对话进行总结摘要，替换原始消息

**跨会话记忆（计划实现）：**
- 文件记忆：`~/.repopilot/memories.md`，记录用户偏好、项目结构、常见问题
- 异步总结：会话结束后后台任务（jobs.sqlite队列）用fast模型提取关键信息
- 记忆注入：新会话开始时将memories.md的内容注入system prompt
- 记忆类型：
  - **事实记忆**：用户偏好（语言、框架）、项目结构
  - **经验记忆**：之前解决过的类似问题及方案
  - **错误记忆**：犯过的错误，避免重复

**为什么用文件存储而非向量数据库？**
- 单用户场景，记忆量小（memories.md通常几KB到几十KB）
- 直接注入system prompt比向量检索更可靠（不会漏掉重要信息）
- Claude Code就是用这种方式：`CLAUDE.md`文件存储项目指令

### Q3: 长期记忆怎么处理？异步总结怎么实现？

**长期记忆设计：**

```
用户消息进来
    ↓
[会话内循环] ReAct Loop
    ↓
会话结束/达到触发条件
    ↓
[异步总结] 后台任务（不阻塞用户）
    ↓
提取关键信息：
  - 项目结构变化
  - 用户编码偏好
  - 解决的bug和方案
  - 常用命令和配置
    ↓
合并到 memories.md（去重、归类）
    ↓
下次会话启动时注入system prompt
```

**异步实现方案：**
- 用SQLite做任务队列（jobs.sqlite），存储待总结的会话ID
- CLI退出前fork一个后台进程（或用threading daemon线程）处理总结
- 总结prompt设计："从以下对话中提取：1.用户编码偏好 2.项目关键信息 3.值得记住的解决方案 4.错误教训"
- 增量合并：读取现有memories.md，LLM判断新信息是否重复，合并更新
- 记忆也有上限，超过阈值用LLM做二次压缩

### Q4: 沙盒隔离具体怎么做的？安全机制有哪些？

**Docker沙盒实现细节：**

1. **容器隔离**
   - 每个Agent会话启动一个独立Docker容器（`python:3.10-slim`镜像）
   - 容器用`sleep infinity`保活，所有命令通过`docker exec`执行
   - 会话结束自动kill和remove容器，不留残留

2. **资源限制（cgroups）**
   - `mem_limit=2g`：内存最多2GB，超出OOM kill
   - `cpu_quota=200000`（cfs_period_us=100000）：限制2个CPU核心
   - 防止Agent执行死循环或内存炸弹

3. **文件系统隔离**
   - bind mount：只将目标仓库目录挂载到容器`/workspace`
   - 容器内只能看到/workspace，无法访问宿主其他目录
   - 注意：bind mount是读写的，容器内修改会反映到宿主（这是必要的，因为Agent要改代码）
   - LocalSandbox额外有`_safe_path()`路径穿越检查：用`Path.relative_to()`确保解析后的路径在repo根目录内

4. **网络隔离**
   - `network_mode`可配置为`bridge`（可访问外网）或`none`（完全断网）
   - 安装依赖时用bridge，执行不信任代码时切到none

5. **执行超时**
   - 每个exec命令默认30秒超时
   - Docker SDK没有原生exec超时，手动用流式读取+时间判断实现超时中断

6. **Shell注入防护**
   - 所有传给shell的用户输入用`shquote()`转义（单引号包裹，处理单引号嵌套）
   - 防止Agent构造恶意命令如`; rm -rf /`

**权限模式（approval_mode）：**
- `auto`：自动批准所有非黑名单操作（完全信任，--danger-full-access）
- `confirm`：真正安全的只读命令自动放行，代码执行/网络/写操作/环境变量一律询问用户（默认，对齐Claude Code/Codex CLI）
- `edit-only`：只读操作自动放行，写操作需确认，所有exec命令拒绝
- `deny`：只读模式，禁止任何修改和执行

**Permission Engine 核心设计（T6实现）：**

Permission Engine 是工具调用前的第一道门，核心原则：**安全黑名单永远拒绝，白名单极端保守，其余询问用户**。

*硬拒绝（Deny，任何模式都不允许）：*
- `rm -rf /`、`sudo`、`curl|sh`管道执行、`chmod -R 777`、`git push --force`、fork炸弹、`crontab`、`reboot/shutdown`
- `base64 -d | sh`等编码绕过、`source ~/.bashrc`等shell注入
- 写/读敏感路径：`~/.ssh/*`、`.env`、`/etc/*`、`~/.aws/`、`*.pem`、shell初始化文件
- 命令中引用敏感路径（`cat ~/.ssh/id_rsa`、`cat /etc/passwd`）
- network=none时拒绝curl/wget/pip install/git clone等网络命令
- bash中`cat ../`路径穿越workspace

*自动放行（Allow，仅confirm模式下真正安全的命令）：*
- 纯只读、无网络、无副作用、无代码执行：`ls/cat项目文件/head/tail/grep/find/wc/sed(pure)/echo/pwd`
- `git status/diff/log`（只读子命令）、`pytest/ruff/mypy/black`等测试/lint工具
- 所有read-only工具（read_file/grep/glob/list_dir/get_repo_tree）在所有模式下都allow

*询问用户（Ask，confirm模式下需要确认）：*
- 代码执行：`python -c`、`node -e`、任何解释器的-c/-e/--eval flag
- 网络命令：curl/wget/pip install/npm install/git push（network enabled时）
- 环境变量泄露：`env`、`printenv`
- 写操作：write_file/edit_file
- 所有不在白名单也不在黑名单的命令（默认保守策略）

*Always-Allow记忆：* 用户选[a]lways后，本session内相同(tool, arg)自动放行。

*安全审计经验：* 白名单不能只看命令前缀——`python`是白命令但`python -c`执行任意代码；`cat`是白命令但`cat /etc/passwd`读系统文件；`echo`是白命令但`echo x > file`写文件。需要检查：code-exec flags(-c/-e)、write operators(>/>>/| tee)、sensitive path fragments、git subcommand类型。

### Q5: 为什么用 LiteLLM？模型切换怎么实现的？

**选择LiteLLM的原因：**
- 统一了100+LLM提供商的API格式（OpenAI/Anthropic/火山/DeepSeek/GLM等）
- 调用接口完全兼容OpenAI SDK，切换模型只需改一个字符串
- 自动处理各提供商的参数差异（`drop_params=True`忽略不兼容参数）
- 支持重试、fallback、成本统计等高级功能

**模型路由策略：**
```python
self.models = {
    Tier.FAST:    "openai/doubao-seed-1-6-flash-250828",   # 15s超时
    Tier.DEFAULT: "openai/doubao-seed-evolving",            # 25s超时，5M上下文
    Tier.STRONG:  "openai/doubao-seed-2-1-pro-260628",      # 40s超时
}
```
- 规划/反思/压缩/分类任务用fast模型（省成本、速度快）
- 主要编码循环用default模型（5M上下文足够装下大代码库）
- 最终答案生成用strong模型（质量最高）
- 用户可以通过CLI命令`repopilot model <name>`随时切换，或在config.toml中配置
- API Key配置：支持在config.toml中设置，也支持从环境变量`OPENAI_API_KEY`/`ARK_API_KEY`读取

### Q6: 熔断器是怎么实现的？为什么需要它？

**为什么需要熔断器：**
- LLM API可能出现持续故障（API Key过期、额度用完、服务宕机）
- 如果没有熔断，每次调用都会等待超时（25-40秒），50步循环就是20分钟的无效等待
- 熔断器能快速失败，避免资源浪费和雪崩效应

**滑动窗口熔断器实现：**

三种状态转换：
```
         失败率>=50%（窗口内>=5次调用）
    Closed ─────────────────────────→ Open
      ↑                               ↓
      │ 探测成功                     │ 等待60秒冷却
      │                               ↓
    Half-Open ←──────────────────────┘
         探测失败→回到Open
```

**参数设计：**
- `window=20`：只看最近20次调用的结果（滑动窗口，用`collections.deque(maxlen=20)`实现）
- `min_calls=5`：至少5次调用才开始判断（样本太少不触发）
- `error_rate=0.5`：50%失败率触发熔断（不是100%，容忍偶发错误）
- `cooldown=60s`：熔断后60秒进入半开状态，放一个探测请求
- 用`threading.Lock()`保证线程安全

**重试策略配合：**
- 单次调用层面：指数退避重试（base=0.5s, cap=8s），全抖动（`delay * random(0.5, 1.0)`）
- 重试只针对可重试错误：超时/连接错误/429限流/5xx
- 不重试：400(请求格式错)/401(认证失败)/403(权限)/404(模型不存在)
- 全局层面：熔断器在多次重试仍然失败后触发，快速拒绝后续请求

### Q7: 工具系统怎么设计的？Agent怎么知道调用哪个工具？

**工具定义（OpenAI function calling格式）：**

每个工具是一个JSON Schema描述的函数：
```json
{
  "type": "function",
  "function": {
    "name": "read_file",
    "description": "Read a file with line numbers. Use offset/limit for large files.",
    "parameters": {
      "type": "object",
      "properties": {
        "path": {"type": "string", "description": "File path relative to repo root"},
        "offset": {"type": "integer", "default": 0, "description": "Start line (0-based)"},
        "limit": {"type": "integer", "default": 200, "description": "Max lines to read"}
      },
      "required": ["path"]
    }
  }
}
```

**当前实现的工具集：**

| 工具 | 功能 | LLM何时使用 |
|------|------|------------|
| `read_file` | 读文件（带行号，支持分页） | 需要看文件内容时 |
| `write_file` | 创建/覆盖写文件 | 创建新文件时 |
| `edit_file` | 精确字符串替换（返回diff） | 修改已有代码时（优先于write_file） |
| `exec` | 执行shell命令（超时控制） | 运行测试/构建/安装依赖时 |
| `glob` | 文件名模式匹配 | 找文件位置时（`**/*.py`） |
| `grep` | 正则搜索内容 | 查找函数定义/变量引用时 |
| `list_dir` | 列目录树 | 了解目录结构时 |
| `get_repo_tree` | 仓库代码地图 | 初始化时了解项目结构 |

**Agent调用流程（ReAct Loop）：**
1. System Prompt中描述每个工具的用法和策略（如"优先用edit_file而非write_file修改代码"）
2. 将tools数组传给LLM的`tools`参数
3. LLM返回`tool_calls`（可能一次返回多个工具调用，并行执行）
4. Agent通过Sandbox层执行工具，收集结果
5. 将工具结果作为`tool`角色消息追加到messages
6. 重复以上步骤直到LLM返回纯文本（无tool_calls），即最终答案

**关键设计原则：**
- 工具返回结果要有好的格式（带行号、truncated提示、exit code），方便LLM理解
- 长输出自动截断（head 500字 + tail 1500字），防止浪费token
- 工具错误信息要友好，给LLM修正的线索（如edit_file找不到字符串时用difflib给最接近匹配）

### Q8: 上下文管理怎么做？Token超限怎么处理？

**三层防护：**

1. **输入截断**
   - `read_file`支持offset/limit分页，LLM先读前200行，需要更多再继续读
   - `exec`输出自动truncate（head 500 + tail 1500字符）
   - `get_repo_tree`限制最多4000 token的输出

2. **预算控制**
   - `budget_tokens=200000`：总token预算，超过就停止
   - `max_steps=50`：最大步数，防止无限循环
   - 每步统计token使用量（从LLM响应的usage字段获取）

3. **自动压缩（Compaction）**
   - 当上下文使用率达到`compact_micro_ratio=0.7`时触发微压缩：用fast模型总结最早的几轮对话
   - 达到`compact_auto_ratio=0.85`时强制压缩：大幅压缩历史，保留最近`recent_keep_steps=10`步
   - 压缩prompt："将以下对话历史总结为简洁的要点，保留关键决策、文件位置和未解决问题"
   - 压缩后：system prompt + 压缩摘要 + 最近N轮完整对话 + 当前任务

**为什么需要压缩而不是直接截断？**
- 直接截断会丢失关键上下文（如之前已经读过的文件、做过的修改决策）
- LLM总结可以保留语义信息，丢弃冗余的工具输出
- 这也是Claude Code和Codex CLI的做法（Claude称auto-compact，Codex称compaction）

### Q9: Code Agent 相比传统RAG有什么优势？代码检索为什么不用向量数据库？

**核心区别：**

| 维度 | 传统RAG | Code Agent（RepoPilot） |
|------|---------|------------------------|
| 检索方式 | embedding向量相似度（近似） | grep/glob/tree-sitter（精确） |
| 检索结果 | top-K最相似片段 | 精确匹配所有结果 |
| 幻觉风险 | 可能漏掉关键片段 | grep是确定性的，不会漏 |
| 交互模式 | 一问一答 | 多轮ReAct循环，可迭代探索 |
| 操作能力 | 只能回答问题 | 能读/写/执行代码，闭环完成任务 |
| 验证能力 | 无 | 可执行测试验证修改正确性 |
| 适用场景 | 文档问答/知识库 | 代码编写/修改/调试 |

**为什么代码领域不需要RAG：**
1. 代码是结构化的，grep搜索函数名/类名/变量名是100%精确的，不需要模糊匹配
2. tree-sitter可以解析AST精确找到函数定义、调用关系，比embedding更准
3. 代码在本地，直接访问文件系统比先embedding再检索快且准
4. Agent可以通过glob+grep+read组合探索，类似人类程序员的工作方式——先ls找到文件，再grep找定义，再read看细节
5. 向量检索有召回率问题，可能漏掉关键代码，在代码场景下不可接受

**什么时候RAG有用：**
- 多租户SaaS场景（多个用户共享一个知识库服务）
- 非代码文档（PDF、Wiki）的语义检索
- 大规模代码库（百万文件级别）时可以作为辅助索引

### Q6: Repo Map 是怎么实现的？为什么用 tree-sitter 而不是正则或 LSP？

**实现细节：**

Repo Map 是给 LLM 的代码结构概览，相当于给Agent一张"代码地图"，让它不用读所有文件就能知道类/函数在哪里。

*为什么选 tree-sitter：*
- 正则表达式无法处理嵌套结构（类里的方法、装饰器、嵌套函数）
- LSP太重（需要启动language server进程，初始化慢，依赖多）
- tree-sitter是增量解析器，解析Python只需几十毫秒，无运行时依赖（纯C编译的绑定）
- 支持多语言（后续加JS/TS/C++只需装对应的language包，代码无需改）

*实现流程：*
1. `iter_source_files()`遍历仓库，用`pathspec`应用.gitignore规则，跳过二进制和忽略文件
2. 对每个.py文件，`index_file()`读取源码，tree-sitter Parser生成AST
3. `_walk()`递归遍历AST节点：
   - `class_definition` → 提取class name、docstring，递归进block找方法
   - `function_definition` → 提取function/method name、docstring（通过doc_of找block里第一个string literal）
   - `decorated_definition` → 处理@decorator包装的类/函数
   - `import_statement`/`import_from_statement` → 提取import文本（限80字符内）
4. 按文件大小从小到大排序（小文件通常是核心定义，大文件往往是依赖/生成代码）
5. 在token预算内组装输出，每文件格式：`rel/path.py` → `  L行号  def/class 名称 — docstring`
6. 超过预算时停止，显示"N more files omitted"

*关键参数：* `CHARS_PER_TOKEN=4`、`max_tokens=4000`（默认）、大文件阈值`100_000`字节、import截断80字符、symbols上限30个/文件。

### Q7: Session Store 为什么用 JSONL + SQLite 而不是纯JSON或纯SQLite？

**设计考量：**

- **JSONL的优势**：append-only写入，每行独立JSON，写入崩溃不会损坏整个文件（之前写入的行都是完整的）；可以用grep/jq/编辑器直接查看；流式写入不需要读入整个文件
- **SQLite索引的优势**：快速列出所有会话（ORDER BY updated_at DESC LIMIT N），不需要打开每个JSONL文件；存储元数据（title/cwd/model/tokens_used）用于会话列表UI
- **为什么不用纯JSON**：JSON数组追加需要解析整个文件→加元素→重写整个文件，会话很长时性能差，崩溃可能损坏整个文件
- **为什么不用纯SQLite**：大文本（tool result可能几KB到几十KB）存在SQLite里性能差、难以人工查看、迁移困难；SQLite是追加写吗？不，SQLite的B-tree是随机写，并发写入需要WAL模式配置

*rewind实现细节：*
- 读全部JSONL行→取前N个→覆盖写文件→更新SQLite last_event_id
- 这是"截断重写"模式，rewind是低频操作（用户手动触发/Agent纠错触发），不需要append-only高性能
- rewind后append_event时，_count_lines()重新数文件行数作为next_id，保证ID连续

### Q8: 三级上下文压缩具体怎么实现的？阈值和参数是什么？

**三级压缩的设计理念：渐进式，能用规则就不用LLM，能少压缩就不多压缩。**

**第一级：tool-compact（规则，零成本）**
- 时机：每个工具结果存入Context前自动调用
- 逻辑：如果输出超过12000字符或200行，保留前2/3头部和后1/6尾部，中间插入"N chars/lines omitted"
- 为什么是head+tail而不是只保留head？工具输出的结尾通常有总结信息（测试结果、错误堆栈的最底层），也很重要
- 参数：`max_chars=12000`, `max_lines=200`

**第二级：micro-compact（fast LLM，75%阈值）**
- 时机：`token_usage_ratio() >= 0.75`
- 逻辑：取最老的3-5步（如果总步数>10），用fast模型把它们压缩成1-2句总结，拼到summary字段，从steps列表移除压缩的步
- Prompt设计：要求保留"尝试了什么、关键结果/错误、相关文件路径和符号"，限制80词以内
- 这是轻量级压缩，只压缩少量最早的步骤，成本低（fast模型几十token输入+几十token输出）

**第三级：auto-compact（fast LLM，90%阈值）**
- 时机：`token_usage_ratio() >= 0.90`
- 逻辑：把除了最近10步（RECENT_STEPS_KEEP=10）之外的所有历史用fast模型生成结构化摘要，替换整个summary字段，steps列表只保留最近10步
- 结构化摘要包含4个section：Completed Actions / Key Findings / File Locations / Unresolved
- 这是重度压缩，在上下文快溢出时触发，牺牲细节保留全局信息

*Token估算校准：*
- 初始估算用`len(text) // 4`（中英混合代码近似值）
- 每次拿到LLM响应的真实`usage.prompt_tokens`，调用`update_actual_usage(actual_tokens)`计算校准系数
- 后续估算乘以校准系数，越来越准确

*Token预算分配（以默认120K budget为例）：*
- System: ~1K tokens
- Project Context (repo map + memory + skills): ~5K
- Plan: ~1K
- Summary: ~2K
- Recent Steps: ~111K（剩余全部给最近对话）
