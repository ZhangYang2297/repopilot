# RepoPilot 核心知识与面试指南

> 本文档用最简单的话讲清楚整个项目，每个知识点都有例子，面试时直接用。
> 记住：面试官不关心你用了什么框架，关心的是"为什么这么做"和"你解决了什么问题"。

## 一、30秒电梯演讲（开口就赢）

**面试官：用一句话介绍你的项目。**

> RepoPilot 是一个本地优先的 Code Agent CLI 工具，对标 Claude Code 和 OpenAI Codex CLI。
> 你在终端 `cd` 到项目目录，输入 `repopilot`，然后用自然语言告诉它要做什么——比如"修复这个bug"、"给这个函数加类型注解"、"跑一下测试然后修失败的"，它会自动读代码、搜索、编辑、跑命令、验证，最后给你结果。
> 核心亮点：纯 ReAct 架构（单模型循环，不搞复杂的多agent编排）、Docker沙箱隔离、四层权限控制、分层记忆系统、12/12 E2E任务成功率。

## 二、架构总览（画图讲最清楚）

**面试官：整体架构是怎样的？画一下。**

你在白板上画这个（从下往上）：

```
┌─────────────────────────────────────────────┐
│  CLI 层 (Typer + Rich)     REPL / 单次任务   │  用户交互
├─────────────────────────────────────────────┤
│  Agent Loop (ReAct)        Think→Act→Observe │  大脑
├─────────────────────────────────────────────┤
│  Context Manager           L0-L5 分层记忆     │  短期/长期记忆
├─────────────────────────────────────────────┤
│  Tool Registry             10个工具+权限校验  │  手
├─────────────────────────────────────────────┤
│  Permission Engine         auto/confirm/...  │  安全门
├─────────────────────────────────────────────┤
│  Sandbox 层                Local / Docker    │  执行环境
├─────────────────────────────────────────────┤
│  LLM Service               LiteLLM统一封装   │  模型调用
├─────────────────────────────────────────────┤
│  配置层                    TOML + 环境变量    │  配置
└─────────────────────────────────────────────┘
```

**一句话解释每层（背下来）：**
- **CLI层**：接收用户输入，Typer做命令解析，Rich做彩色输出。直接 `cd project && repopilot` 就进入交互模式
- **Agent Loop**：核心循环，就是"想→调工具→看结果→再想"，纯 ReAct 模式，不搞 Plan-Act-Reflect 多模型，因为 Claude Code/Codex 都是这么做的——强模型自己就能推理
- **Context Manager**：管理发给LLM的消息，有分层记忆（下面记忆系统专题讲），还有三级自动压缩防止token爆掉
- **Tool Registry**：注册所有工具（读文件、写文件、grep搜索、glob找文件、list_dir列目录、bash执行命令、run_python跑Python、edit_file精准编辑、finish结束），每次调用前经过Permission Engine检查
- **Permission Engine**：4种模式——auto（全自动，适合可信项目）、confirm（危险操作弹y/N让用户确认）、edit-only（只能读和编辑文件，不能执行命令）、deny（只读模式）。内置黑名单（rm -rf、curl|sh、写~/.ssh等）
- **Sandbox层**：两种实现——LocalSandbox直接操作本地文件（快，适合信任的项目），DockerSandbox在容器里执行（安全，cgroups限制CPU/内存，可选禁网）。所有路径操作经过 `_safe_path()` 校验防止穿越
- **LLM Service**：通过LiteLLM调用任何OpenAI兼容API，有熔断器（连续失败5次打开，30s后半开重试）、指数退避重试（1s→2s→4s→8s）、三档超时（fast=30s/default=120s/strong=300s）

## 三、记忆系统（高频面试题）

### Q：Agent的记忆管理具体怎么做？处理流程是什么？

这是最常问的题。**分两层回答：会话内记忆 + 持久记忆。**

**第一层：会话内记忆（Context Manager，L0-L5分层）**

每次调用LLM时，消息按以下顺序组装：

| 层 | 内容 | Token大小 | 什么时候用 |
|----|------|-----------|-----------|
| L0 | System Prompt（系统指令）| ~1.2K | 每次必带 |
| L0.5 | 记忆层（全局+项目+session notes）| 几百到1-2K | 每次必带 |
| L2 | 压缩摘要（旧步骤的总结）| ~1-2K | token到75%时自动生成 |
| L4 | 最近10步完整消息 | 填满剩余预算 | 每次必带 |
| L5 | 最新工具结果 | 随对话动态变 | 每次必带 |

**举个具体例子：**
```
你说："main.py里add函数有bug，帮我修"
→ L0: "你是RepoPilot，一个代码助手..."
→ L0.5: "项目记忆：这个项目用pytest跑测试；全局记忆：回复要简洁"
→ L4[最近消息]: 你说的"修bug"这句话
→ 模型回复：调用read_file("main.py")
→ 执行read_file，拿到文件内容 → 进入L5
→ 下一轮调用LLM时，L4包含了read_file的调用和返回结果
→ 又调用edit_file修改bug
→ 又调用bash跑pytest
→ ...循环直到任务完成
```

**三级压缩是什么？为什么需要？**

因为大模型的上下文窗口是有限的（即使用5M token的模型，全塞进去也费钱且注意力会分散），所以：

1. **tool-compact（每次工具返回时）**：bash命令的输出如果超过2000头+5000尾字符，截断中间部分，显示"跳过了XXX字符"。这步是即时的，不调LLM。
2. **micro-compact（token用到75%时）**：用fast模型把最旧的3-5步对话总结成一段话，替换原始消息。比如原来10条消息"读文件→内容xxx→编辑→diff→跑测试→输出xxx"压缩成"读取了main.py，修复了add函数的符号错误，pytest通过"。
3. **auto-compact（token用到90%时）**：除了最近10步保留原始消息，其他全部用fast模型压缩成摘要。

**举个形象比喻：**
- 你在做数学题，tool-compact就是计算器显示不下了只保留开头和结尾的数字
- micro-compact就是你做题做了一半，翻前面的草稿纸，把前几步的计算过程在脑子里归纳成"已经算出x=3"
- auto-compact就是你做了一下午题，把前面所有草稿纸收起来只写一句"前面的结论是xxx"，只保留最近的几道题

**第二层：持久记忆（文件驱动，不使用向量数据库）**

这里对标Claude Code的CLAUDE.md和Codex的AGENTS.md。

**为什么不用RAG/向量数据库？**
1. **代码本身就是可检索的**——grep/glob/tree-sitter是确定性检索，想找什么直接搜，比embedding精确
2. **单租户CLI场景不需要**——用户每次只在一个项目里工作，代码量最多几万文件，grep一秒搜完
3. **人可读可git**——记忆就是一个Markdown文件，用户直接编辑，也可以提交到git跟团队共享
4. **不需要额外依赖**——不用装ChromaDB、不用embedding模型、不用构建索引

持久记忆分两层：

| 层级 | 文件位置 | 内容 | 示例 |
|------|---------|------|------|
| 全局记忆 | `~/.repopilot/REPOPILOT.md` | 用户个人偏好 | "用中文回复"、"回答要简洁" |
| 项目记忆 | `<repo>/REPOPILOT.md` | 项目约定 | "测试命令是pytest -v"、"用类型注解"、"不要修改migrations/目录" |

**启动时自动加载**：进入REPL时自动读这两个文件，注入到L0.5层。如果不存在，全局记忆首次运行时创建模板。

**/memory命令**：
- `/memory` 显示当前加载的记忆
- `/memory 注意要先跑测试` 追加一条到项目记忆
- `/memory --global 用中文回复` 追加到全局记忆

**跨会话恢复（/resume）**：
所有对话事件（用户消息、assistant回复、工具调用、工具结果）都以JSONL格式保存到 `~/.repopilot/sessions/YYYY/MM/DD/rollout-<timestamp>-<id>.jsonl`，SQLite存索引。
`/resume` 读取最近一个JSONL，把user_msg/assistant_msg/tool_result事件重新回放进ContextManager，就像对话从未中断。

**举个跨会话例子：**
```
$ repopilot          # 第一次进入
repopilot> 记住我最喜欢的语言是Rust
repopilot> /exit

$ repopilot          # 第二天重新进入
repopilot> /resume   # 恢复昨天的对话
repopilot> 我最喜欢的语言是什么？
Rust                # 模型从恢复的上下文中正确回答
```

### Q：Session Store 怎么设计的？为什么用 JSONL + SQLite？

**为什么两个存储？**

- **JSONL**（每行一个JSON）存实际事件数据。好处：追加写入（append-only），写入快；文件损坏只丢最后一行；可以直接用cat/grep查看；跨工具可读。
- **SQLite**存索引（session id、标题、cwd、模型、创建时间、token用量、最后事件id）。好处：按时间查询、列出最近会话、按id查找session路径很快。

**举个例子：**
```
~/.repopilot/sessions/2026/07/13/rollout-1720876800-abc123def456.jsonl
{"id":1,"ts":"2026-07-13T12:00:00Z","type":"user_msg","payload":{"content":"fix the bug"}}
{"id":2,"ts":"...","type":"assistant_msg","payload":{"content":"Let me read main.py","tool_calls":[...]}}
{"id":3,"ts":"...","type":"tool_call","payload":{"tool":"read_file","args":{"path":"main.py"}}}
{"id":4,"ts":"...","type":"tool_result","payload":{"content":"def add...","call_id":"..."}}
```

SQLite里的threads表记录：id=abc123, title="fix the bug", cwd="/path/to/repo", model="openai/doubao-seed-evolving", created_at=..., last_event_id=4。

## 四、ReAct Agent Loop（核心）

### Q：Agent的思考流程是什么？用的什么模式？

**纯ReAct循环，不是Plan-Act-Reflect，不是多Agent。**

**为什么？** 因为我们参考了Claude Code和Codex CLI的实际实现——它们就是纯ReAct。强模型（doubao-seed-evolving / Claude 4 / GPT-4o）的推理能力足够强，不需要单独的Planner模型来生成计划。多模型编排反而增加延迟、复杂度和出错点。

**循环流程（伪代码）：**
```python
while steps < max_steps:
    # 1. 检查是否需要压缩上下文
    if ctx.token_usage() > 90%:
        ctx.auto_compact(llm)
    elif ctx.token_usage() > 75%:
        ctx.micro_compact(llm)

    # 2. 组装消息，调用LLM
    messages = ctx.build_messages()
    response = llm.chat(messages, tools=tool_schemas)

    # 3. 解析回复：文本 or 工具调用 or finish
    parsed = parse_response(response.content, response.tool_calls)

    if parsed.is_finish:
        print(response.content)
        break

    if parsed.has_tool_calls:
        for tc in parsed.tool_calls:
            # 4. 权限检查
            decision = permission_engine.check(tc.name, tc.args)
            if decision == "deny":
                ctx.add_tool_result(tc.id, "Permission denied")
                continue
            # 5. 执行工具
            result = tool_registry.execute(tc.name, tc.args, sandbox)
            # 6. 结果加入上下文
            ctx.add_tool_result(tc.id, result.content)
    else:
        # 纯文本回复 = 最终答案
        ctx.add_assistant(response.content)
        print(response.content)
        break
```

**Parser怎么解析模型输出？**
- 优先使用OpenAI原生的`tool_calls`字段（模型直接返回结构化函数调用）
- 如果模型输出了XML格式的工具调用（`<tool_call>...</tool_call>`），用XML parser兜底解析
- 纯文本没有工具调用 → 这就是最终回答

## 五、工具系统

### Q：有哪些工具？工具注册和执行机制是怎样的？

**10个核心工具：**

| 工具 | 功能 | 安全级别 |
|------|------|---------|
| read_file | 读文件，显示行号 | 只读，总是允许 |
| write_file | 写/创建文件（全量覆盖） | 写操作 |
| edit_file | 精准替换（old_string→new_string） | 写操作，最安全 |
| grep | 正则搜索（跨文件） | 只读 |
| glob | 文件名模式匹配（`**/*.py`） | 只读 |
| list_dir | 列目录树（max_depth控制深度） | 只读 |
| get_repo_tree | tree-sitter代码结构地图 | 只读 |
| bash | 执行shell命令 | 执行，最危险 |
| run_python | 执行Python脚本（临时文件） | 执行 |
| finish | 标记任务完成 | 元操作 |

**edit_file的安全设计（重点）：**
- old_string必须至少3个有效字符（防止匹配空字符串误替换）
- old_string在文件中必须恰好出现1次（多次匹配会报错，要求提供更多上下文）
- 原子写入：先写临时文件，然后`os.replace()`原子替换（不会出现半写入的损坏文件）
- 编辑前用difflib找最接近的匹配，如果找不到给出提示"Closest match: ..."

**举个edit_file例子：**
```
old_string: "def add(a, b):\n    return a - b"
new_string: "def add(a, b):\n    return a + b"
→ 如果old_string出现0次：报错"old_string not found，closest match: ..."
→ 如果出现2次：报错"appears 2 times, include more context"
→ 恰好1次：原子替换，返回unified diff
```

**bash工具的优化：**
- 默认超时120秒，最大600秒（支持跑pytest全量、pip install等长任务）
- Windows自动翻译常用Unix命令（ls→dir、cat→type、rm -rf→rmdir /s /q等）
- 跨平台提示注入system prompt（告诉模型当前是Windows/Linux，用对应命令）
- 输出截断：head=2000字符 + tail=5000字符（防止pip install输出几万token）

## 六、沙箱与安全

### Q：沙箱怎么实现的？怎么保证安全？

**两种沙箱：Local和Docker。**

**LocalSandbox（直接在本地操作）：**
- 所有路径经过`_safe_path()`校验：`(repo_path/user_path).resolve().relative_to(repo_path)`，如果resolve后跑出了repo_path就抛PermissionError
- **举例**：用户尝试`read_file("../../etc/passwd")`→resolve后变成`/etc/passwd`→`relative_to(repo_path)`抛ValueError→PermissionError拒绝
- write_file用原子写：tempfile + os.replace，防止写入中断损坏原文件
- exec()用subprocess.run(shell=True, cwd=repo_path, timeout=...)，cwd被_safe_path限制在repo内

**DockerSandbox（容器内执行）：**
- 启动一个Python 3.10容器，把repo目录mount进容器（只读或可读写可配置）
- cgroups限制：CPU 2核、内存2G
- 网络可选：bridge模式（可联网）或none（完全断网，防止数据外传）
- 容器内执行exec命令，通过Docker SDK传回来stdout/stderr
- 用完自动删除容器

**Permission Engine（安全门）：**

| 模式 | 读文件 | 写/编辑文件 | 执行命令 |
|------|--------|-----------|---------|
| auto | 允许 | 允许（黑名单除外）| 允许（黑名单除外）|
| confirm | 允许 | 弹确认 | 弹确认 |
| edit-only | 允许 | 允许 | **拒绝** |
| deny | 允许 | **拒绝** | **拒绝** |

**黑名单规则（所有模式下强制拒绝）：**
- `rm -rf /`、`del /S /Q C:\*`（破坏性删除）
- `curl ... | sh`、`wget ... | bash`（管道执行远程脚本）
- 写入`~/.ssh/`、`~/.aws/credentials`（窃取凭证）
- `sudo`、`su`（提权）
- `git push --force`（强制推送）

**举个安全场景：**
```
用户用confirm模式启动repopilot
Agent尝试执行: bash("rm -rf build/")
→ Permission Engine检查：rm -rf但不是系统根目录，路径在repo内
→ 弹出提示："允许执行 rm -rf build/? [y/N/e/d]"
→ y=允许一次，N=拒绝，e=编辑命令后执行，d=总是允许这个命令
```

## 七、LLM Service

### Q：怎么做模型调用？为什么用LiteLLM？熔断器、重试怎么设计的？

**为什么用LiteLLM？**
它是一个统一的LLM调用库，把OpenAI、Anthropic、火山引擎、DeepSeek、智谱等所有API都统一成OpenAI格式。这样用户只需要换一个model参数就能切换模型，代码不需要改。

**三档模型设计：**
- FAST tier（30s超时）：便宜快速的模型，用来做压缩、总结等简单任务
- DEFAULT tier（120s超时）：主力编码模型
- STRONG tier（300s超时）：复杂推理（当前三档默认都是同一个模型，因为实际使用中用户往往只用一个模型，但架构支持分离）

**熔断器（Circuit Breaker）：**
```
CLOSED（正常）→ 连续5次LLM调用失败 → OPEN（熔断）
OPEN（熔断）→ 立即返回错误，不调API → 等待30秒
HALF_OPEN（半开）→ 放一个请求试探 → 成功→CLOSED，失败→OPEN
```
**为什么需要熔断器？** 如果API key错了或者服务挂了，不应该每次都去等30秒超时再报错，应该快速失败，节省时间。

**指数退避重试（Exponential Backoff）：**
- 第1次失败：等1秒重试
- 第2次失败：等2秒
- 第3次失败：等4秒
- 第4次失败：等8秒
- 最多重试4次
- 只对可重试的错误重试（Timeout、ConnectionError、RateLimitError），不对AuthError（key错了）重试，因为重试也没用

## 八、评测体系

### Q：怎么验证Agent的能力？评测是怎么做的？

**三层测试：**

**1. 单元测试（434个）**
每个模块独立测试：
- 配置加载、环境变量覆盖、TOML持久化
- LLM解析、重试逻辑、熔断器状态转换
- Local/Docker沙箱的文件操作、路径穿越检测
- Permission Engine4种模式、黑名单、白名单
- 工具read/write/edit/grep/glob/bash的正确执行和边界
- Context Manager的消息组装和三级压缩
- Session Store的JSONL+SQLite读写
- 安全修复的回归测试

**2. E2E任务评测（12/12通过，100%成功率）**
自建评测框架eval/run.py，参照SWE-bench范式：
- 每个任务是一个临时目录，setup.py创建有bug的代码，task.md是给agent的自然语言任务，verify.py跑测试验证是否修好
- 12个任务覆盖：单文件bugfix、多bug修复、从零写新功能（TDD）、运行时错误自调试、跨文件修改、安全防护（路径穿越）、大文件边界情况、CLI开发、缺失模块创建、函数补全、正则验证、装饰器重构
- 全自动化：`python -m eval.run`自动跑完所有任务，输出成功率、token用量、耗时统计

**3. 记忆专项评测（8项）**
测试记忆系统的正确性：
- M1 会话内事实回忆
- M2 项目记忆指令遵守
- M3 全局记忆指令遵守
- M5 /clear后上下文隔离
- M6 /memory命令持久化
- M7 压缩后关键信息保留
- M8 多轮信息积累
- M9 跨会话记忆生效

**评测指标：**
- Success@1：单次尝试成功率（最核心的指标，SWE-bench就是用这个）
- 平均步数/任务：反映效率
- 平均token消耗/任务：反映成本
- 平均耗时/任务：反映速度

## 九、为什么不用XXX？（常被挑战的问题）

### Q：为什么不用LangChain/LangGraph/AutoGen？

因为它们太重了。Code Agent的核心逻辑就是一个while循环，LangChain的抽象反而增加了调试难度和依赖。我们只有~5000行代码，LangChain光依赖就几百兆。Claude Code和Codex CLI也没有用这些框架。

### Q：为什么不用RAG/向量数据库？

单租户CLI场景：代码就在本地，grep/glob是确定性检索，比embedding精确；记忆是Markdown文件，人可读可git，不需要向量检索。

### Q：为什么不用多Agent（Planner+Coder+Reviewer三个模型）？

实测发现，强模型（doubao-seed-evolving 5M context）的ReAct单循环效果已经足够好，多Agent增加延迟和token成本，而且在Claude Code/Codex里也是单模型ReAct。三模型架构（规划-执行-审查）是2024年初的做法，现在强模型已经不需要了。
