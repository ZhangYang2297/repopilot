# RepoPilot 项目开发指南

> 版本: v0.1.0 (Step 1–6 完成)
> 本文档记录项目从零到现在的完整开发流程、架构设计、每一步的决策理由和代码结构。

---

## 一、项目定位

**RepoPilot** 是一个本地优先（local-first）的代码 Agent CLI 工具，参考 Codex CLI / Claude Code 的设计理念：

- 单租户场景：每个用户在自己的终端运行，不存在多租户并发，因此不需要向量数据库/RAG服务
- 代码操作直接使用文件系统工具（grep/glob/read/edit/exec），而非嵌入检索
- 所有代码执行在沙盒中运行（Docker 容器隔离 或 Local 受限模式）
- 用户自带 API Key，通过 LiteLLM 统一调用任意 OpenAI 兼容模型（火山引擎 ARK/DeepSeek/GLM 等）
- 三层模型路由（fast/default/strong）平衡成本与质量

## 三、开发步骤回顾（Step 1 → Step 6）

### Step 1: 项目初始化

**做了什么：**
- 使用 `pyproject.toml` 作为项目配置（现代 Python 打包标准，替代 setup.py）
- 安装核心依赖：typer（CLI）、rich（终端美化）、litellm（统一LLM调用）、docker（容器SDK）、pytest（测试）
- 建立包结构 `repopilot/`，设置 `repopilot` 命令入口指向 `repopilot.cli:app`

**关键决策：**
- 选 typer 而非 argparse/click：typer 基于类型注解自动生成 CLI，代码量少且自带帮助文档
- 选 LiteLLM 而非直接用 openai SDK：LiteLLM 统一了 100+ LLM 提供商的调用格式，切换模型只需改模型名
- Python 3.10+：使用 `match`、`X | Y` 联合类型等新语法

**知识点：**
- `pyproject.toml` 是 PEP 621 标准，`[project.scripts]` 定义 CLI 入口点
- `pip install -e .` 以可编辑模式安装，改代码无需重新安装

### Step 2: 配置系统（config.py）

**做了什么：**
- 实现 `Settings` 数据类（dataclass），包含 LLM、Sandbox、Agent、权限、路径等所有配置
- 四层配置加载优先级：**默认值 < config.toml < 环境变量 < 代码覆写**
- TOML 持久化：非默认值才写入配置文件，保持文件简洁
- 首次运行向导：未配置时自动提示用户输入 model/api_key/base_url

**核心字段：**

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `model` | str | "" | 主力模型，LiteLLM格式 `provider/model` |
| `fast_model` | str | "" | 便宜快速模型（规划/反思/压缩） |
| `strong_model` | str | "" | 高质量模型（最终回答） |
| `api_key` | str | "" | API密钥，空则从环境变量找 |
| `base_url` | str | "" | 自定义API地址（火山ARK等） |
| `sandbox_type` | str | "docker" | docker/local |
| `docker_image` | str | "repopilot-sandbox:py310" | 沙盒镜像 |
| `docker_mem_limit` | str | "2g" | 内存限制 |
| `docker_cpu_quota` | int | 200000 | CPU配额（2核） |
| `docker_network` | str | "bridge" | 网络模式（bridge/none隔离外网） |
| `approval_mode` | str | "confirm" | 权限模式：auto/confirm/edit-only/deny |
| `max_steps` | int | 50 | Agent最大迭代步数 |
| `budget_tokens` | int | 200000 | Token预算上限 |
| `tool_timeout` | int | 30 | 工具执行超时秒数 |

**目录结构（~/.repopilot/）：**
```
~/.repopilot/
├── config.toml      # 用户配置
├── sessions/        # 会话历史
├── state.sqlite     # 状态数据库
├── jobs.sqlite      # 后台任务
├── memories.md      # 长期记忆（后续实现）
├── skills/          # 技能目录（后续实现）
├── cache/           # 缓存
└── logs/            # 日志
```

**关键决策：**
- TOML 而非 YAML/JSON：TOML 是 Python 生态标准配置格式（pyproject.toml 用的就是TOML），支持注释，可读性好
- 环境变量映射用 `_ENV_MAP` 字典统一管理，每个变量带类型转换器（int/bool/str）
- `Settings.load()` 中必须先解析 `home_dir`，再读 config.toml，否则读不到正确位置
- API Key 不强制写配置文件，优先从环境变量读取（安全最佳实践）

**知识点与注意点：**
- Python 3.10 没有 `tomllib`（3.11才内置），需用 `tomli` 作为后备
- `dataclass` 的 `field(default_factory=...)` 用于可变默认值（避免所有实例共享同一个对象）
- 单例模式用模块级全局变量 `_settings` + `get_settings()` 缓存，测试时需 `reset_settings_for_tests()`
- 路径中 `~` 需要用 `Path.expanduser()` 展开

### Step 4: LLM 服务层（llm/）

#### 4.1 Circuit Breaker（circuit_breaker.py）

**做了什么：** 实现滑动窗口熔断器模式。

**三种状态：**
- **Closed（闭合）**：正常放行，记录成功/失败
- **Open（断开）**：直接拒绝请求（抛 `CircuitOpenError`），等冷却时间
- **Half-Open（半开）**：冷却时间过后允许一个探测请求，成功则闭合，失败则重新断开

**参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `window` | 20 | 滑动窗口大小（最近N次调用） |
| `min_calls` | 5 | 窗口内最少调用次数才触发熔断判断 |
| `error_rate` | 0.5 | 错误率阈值（50%失败就熔断） |
| `cooldown` | 60秒 | 断开后等待多久进入半开 |

**知识点：**
- 熔断器是微服务架构经典模式（Michael Nygard, Release It!），防止雪崩
- 用 `threading.Lock` 保证线程安全
- 滑动窗口用 `collections.deque(maxlen=window)` 实现，自动淘汰最旧记录
- 全抖动退避（full jitter backoff）：`delay * (0.5 + random() * 0.5)`，防止重试风暴

#### 4.2 LLM Service（service.py）

**做了什么：**
- 统一封装 LiteLLM 调用，支持三层模型路由（Tier.FAST / DEFAULT / STRONG）
- 自动重试：可重试错误（超时、连接错误、429、5xx）自动重试，不可重试错误（401、400、404）直接抛
- 熔断器集成
- 指数退避 + 全抖动
- tool_calls 解析（从 OpenAI 格式响应中提取函数调用）
- 用量统计（token计数）

**三层模型路由：**

| Tier | 超时 | 用途 | 推荐模型 |
|------|------|------|----------|
| FAST | 15s | 规划、反思、压缩、分类 | doubao-seed-1-6-flash |
| DEFAULT | 25s | 主要编码任务 | doubao-seed-evolving (5M上下文) |
| STRONG | 40s | 最终答案、复杂推理 | doubao-seed-2-1-pro / glm-5 / deepseek-v3 |

**错误分类逻辑（`_is_retryable`）：**

| 错误类型 | 是否重试 |
|----------|----------|
| APITimeoutError | ✅ |
| APIConnectionError | ✅ |
| RateLimitError (429) | ✅ |
| BadRequestError (400, 含上下文超限) | ❌ |
| AuthenticationError (401) | ❌ |
| PermissionDeniedError (403) | ❌ |
| NotFoundError (404) | ❌ |
| 5xx 错误 | ✅ |

**知识点：**
- LiteLLM 的 `completion()` 接口和 OpenAI SDK 兼容，但支持所有提供商
- `litellm.drop_params = True` 自动忽略不兼容参数（比如某些模型不支持 temperature）
- API Key 解析优先级：配置文件 > 环境变量（OPENAI_API_KEY > DASHSCOPE_API_KEY > ARK_API_KEY > ANTHROPIC_API_KEY）
- tool_calls 解析时要处理 `arguments` 可能不是合法 JSON 的情况

#### 4.3 Stream Handler（stream_handler.py）

**做了什么：** 为后续流式输出做准备，基于 Rich 的 Live 实现 Markdown 实时渲染。

**事件类型：**
- `text_delta`：文本增量
- `tool_call`：工具调用通知
- `thinking`：思考过程（可选显示）
- `done`：结束

### Step 5: Sandbox 抽象层（sandbox/）

#### 5.1 基类设计（base.py）

**做了什么：** 定义 `Sandbox` 抽象基类，所有Agent操作代码都通过此接口，Local和Docker两种实现可互换。

**统一数据结构：**
- `ExecResult`：命令执行结果（stdout/stderr/exit_code/timed_out/duration_ms）
  - `truncated()` 方法：长输出自动截断为 head(500字) + tail(1500字)，避免撑爆context
- `GrepMatch`：grep结果（file/line_no/content）
- `FileReadResult`：文件读取结果（带行号，支持分页）
  - `offset`/`limit` 支持大文件分页读取
  - 自动加行号格式 `  1|code here`
- `DEFAULT_IGNORE_DIRS`：忽略目录集合（.git/__pycache__/node_modules/.venv等）

**核心接口方法：**

| 方法 | 功能 | 关键参数 |
|------|------|----------|
| `setup()` | 初始化沙盒 | 启动容器等 |
| `teardown()` | 清理资源 | 停止容器等 |
| `read_file(path, offset, limit)` | 读文件（带行号） | offset=起始行(0-based), limit=最大行数 |
| `write_file(path, content)` | 写文件（覆盖） | |
| `edit_file(path, old, new)` | 精确替换 | 返回unified diff |
| `exec(command, timeout, cwd)` | 执行shell命令 | timeout默认30秒 |
| `glob(pattern)` | 文件名模式匹配 | 支持 `**/` 递归 |
| `grep(pattern, glob_filter, ignore_case)` | 正则搜索 | 返回 GrepMatch 列表 |
| `list_dir(path, max_depth)` | 目录树 | 返回嵌套dict |
| `get_repo_tree(max_tokens)` | 仓库概览 | 后续用tree-sitter替换 |

**安全机制：**
- `_safe_path()` 方法：解析用户提供的路径，用 `relative_to()` 确保路径不逃逸出 repo 根目录（路径穿越防护）
- 上下文管理器协议（`__enter__`/`__exit__`）确保资源自动清理

**知识点：**
- 抽象基类用 `abc.ABC` + `@abc.abstractmethod`
- 行号添加：`f"{i:{width}}|{line}"` 动态宽度对齐
- 路径穿越防护是 Code Agent 最基本的安全要求，防止Agent通过 `../../etc/passwd` 访问仓库外文件

#### 5.2 Local Sandbox（local_sandbox.py）

**做了什么：** 直接在本地文件系统操作，用于开发调试和信任场景。

**实现要点：**
- `read_file`：分页读取，带行号，UTF-8 + errors="replace" 容错
- `write_file`：自动创建父目录
- `edit_file`：精确字符串替换 + difflib 辅助找最相似匹配（当old_string找不到时给提示）+ 返回unified diff
- `exec`：subprocess.run + shell=True + timeout，Windows用cmd.exe
- `glob`：用 Path.rglob/gglob 实现，过滤忽略目录
- `grep`：os.walk 遍历 + re 正则搜索，支持glob过滤和忽略大小写
- `list_dir`：递归walk，返回嵌套dict，隐藏文件和忽略目录不显示

**知识点：**
- `subprocess.run(shell=True)` 在Windows必须用，否则内置命令（dir/echo等）不可用
- `difflib.get_close_matches()` 可以帮Agent找到"最接近的匹配"，当edit_file找不到精确字符串时给出有用提示
- 文件遍历用 `dirs[:] = [d for d in dirs if ...]` 在os.walk中原地剪枝，避免进入忽略目录
- 统一路径分隔符为 `/`（跨平台兼容）

#### 5.3 Docker Sandbox（docker_sandbox.py）

**做了什么：** 用Docker容器实现真正的代码执行隔离，这是生产级安全的核心。

**容器配置：**
- 镜像：`python:3.10-slim`（轻量基础镜像）
- 命令：`sleep infinity`（保持容器运行，反复exec）
- 工作目录：`/workspace`
- bind mount：将宿主机repo目录挂载到容器 `/workspace`（读写模式）
- 资源限制：`mem_limit=2g`、`cpu_quota=200000/100000` = 2核
- 网络模式：bridge（默认）或 none（完全隔离外网）

**实现要点：**
- `setup()`：检查镜像→不存在则pull→启动容器→等待running
- `_docker_exec()`：用Docker API的 `exec_create` + `exec_start(stream=True, demux=True)` 流式获取stdout/stderr，手动实现超时
- `read_file`：通过容器内cat命令读取，用tar传输？→ 实际通过mount直接读写宿主机文件
- `write_file`/`edit_file`：因bind mount，直接写宿主机文件即可（容器内可见）
- `glob`：用容器内 `find -name` 命令
- `grep`：用容器内 `grep -rn` 命令，带exclude-dir
- `list_dir`：用容器内 `find` 命令解析成嵌套dict
- `shquote()`：安全shell转义（单引号包裹，处理单引号嵌套）

**teardown：** kill + remove容器，不留残留

**知识点：**
- Docker SDK 的 `containers.run(detach=True)` 启动后台容器，用 `sleep infinity` 保活
- bind mount 让宿主机目录直接映射到容器内，无需COPY文件，实时同步
- `demux=True` 分离stdout和stderr流
- exec_inspect 获取exit code（exec_start流结束后才能拿到）
- `docker.types.Mount` 定义挂载点
- CPU配额：cfs_period=100000（100ms），cfs_quota=200000表示2个CPU周期
- Shell注入防护：所有用户输入传给shell的都用shquote()转义

- Docker镜像加速源不稳定是常态，需要准备多个备选
- Docker Desktop设置文件在 `%APPDATA%\Docker\settings-store.json`

| 14 | 流式输出 | SSE流式渲染 + token实时统计 |
| 15 | Skill 系统 | 可扩展技能（类似Codex的skills） |
